# Vedaz AI Astrologer — Fine-Tuning & Deployment

Fine-tune Qwen2.5 on Vedaz's astrologer chat data and deploy it using vLLM.

## What's in here

```
├── prepare_data.py          # parse messy chat JSON → clean training JSONL
├── finetune_qwen.py         # LoRA fine-tuning with unsloth (runs on free Colab)
├── run_inference.py         # before/after comparison of base vs finetuned
├── vllm_hosting_guide.md    # how to deploy on a VPS with vLLM
├── Chat Data for assessment of applicants.json  # raw chat data (input)
├── train_data.jsonl         # cleaned training data (generated by prepare_data.py)
├── train_split.jsonl        # 85% train split
├── test_split.jsonl         # 15% test split
└── README.md                # you're reading it
```

## Quick Start

### 1. Prepare the data

```bash
pip install -r requirements.txt  # or just: pip install datasets
python prepare_data.py
```

This reads the raw JSON file, handles the mixed formatting (some lines are single-line JSONL, some are multi-line JSON — it's a mess, but the parser handles it), deduplicates, runs safety checks, and outputs clean `train_data.jsonl`.

### 2. Fine-tune (on Colab or GPU machine)

```bash
# install deps (on Colab, use the unsloth installer)
pip install unsloth datasets trl

# run finetuning
python finetune_qwen.py --epochs 3 --base_model "unsloth/Qwen2.5-1.5B-Instruct"
```

This takes about 10-15 minutes on a T4 GPU (free Colab). The LoRA adapter gets saved to `./vedaz-qwen-lora/`.

If you don't have unsloth, the script falls back to regular `transformers` + `peft`. Slightly slower but works the same.

**To use Qwen3 instead:**
```bash
python finetune_qwen.py --base_model "unsloth/Qwen3-1.7B"
```

### 3. Test it

```bash
python run_inference.py --adapter_path ./vedaz-qwen-lora
```

Runs 5 test prompts through both the base model and the finetuned model, prints them side by side so you can see the difference.

### 4. Deploy on VPS

See [vllm_hosting_guide.md](vllm_hosting_guide.md) for the full walkthrough — covers VPS selection, vLLM setup, systemd service, nginx reverse proxy, and cost estimates.

## Design Decisions

**Why Qwen2.5-1.5B?**
It's small enough to finetune on a free Colab T4, runs fast on cheap GPUs for deployment, and still produces good quality Hindi/Hinglish output. Qwen's multilingual training data includes decent Hindi coverage compared to alternatives like Llama.

**Why LoRA instead of full fine-tuning?**
With only 50 training examples, full fine-tuning would catastrophically overfit. LoRA trains ~0.5% of the parameters, which is enough to teach the model our specific voice and safety rules without destroying its general knowledge.

**Why vLLM for hosting?**
It's the fastest open-source option for serving LLMs, has native LoRA support (can hot-swap adapters), and exposes an OpenAI-compatible API so our backend code doesn't need to change.

## What I'd Improve With More Time

- Add more training data (50 examples is bare minimum — 200-500 would be much better)
- Implement a safety classifier as a post-generation filter
- Set up automated evaluation pipeline comparing model versions
- Add streaming support in the inference script
- Quantize model to AWQ/GPTQ for cheaper deployment
